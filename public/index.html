🎓 UNION Attack Tutorial — Follow these steps exactly as an attacker would

A UNION attack allows an attacker to extract data from other tables by appending a second query to the original one.

STEP 1 — Confirm Injection Point

Before extracting data, an attacker first confirms the field is injectable. They do this by entering a single quote ' and observing if the application throws an error.

Like checking if a door is unlocked before trying to open it — you don't kick it down immediately.

STEP 2 — Count the Columns

UNION SELECT requires the same number of columns as the original query. Attackers try ORDER BY 1, ORDER BY 2... or UNION SELECT 1, UNION SELECT 1,2... until an error disappears, revealing the column count.

STEP 3 — Extract the Users Table

Now the attacker knows there are 6 columns. They replace the dummy numbers with actual column names from the users table.

STEP 4 — Extract Secret Configuration

An attacker doesn't stop at users. They probe for other tables containing API keys, secrets, and credentials.

🎓 What you just did — In 4 steps you:

  • Found an injection point
  • Determined the query structure
  • Extracted all user credentials
  • Stolen production secrets

This is exactly what automated tools like sqlmap do in seconds.

🛡️ ONE LINE fixes all of this:
Use prepared statements / parameterized queries.

🔗 How Reflected XSS is Delivered

Attackers craft a malicious URL and trick victims into clicking it:
https://quicksearch.com/search?q=<script>steal_cookies()</script>
This URL is often hidden using URL shorteners or sent via phishing emails. The victim clicks what looks like a legitimate search link.

URL Builder

⚠️ In a real attack, this URL would be sent to victims via email, social media, or SMS. Never use real XSS payloads on real websites.
🌐 DOM-Based XSS is different — the server is never involved. The attack lives entirely in the URL fragment (the # part) and is processed by JavaScript running in the browser itself. This makes it invisible to server-side security filters.

Fake Welcome Page

socialapp.com/welcome#name=Alice
Welcome, Alice!

This page uses JavaScript to read the #name= part of the URL and display it. The server sends the same page to everyone — the personalization happens in the browser.

// VULNERABLE CODE const name = location.hash.slice(6); document.getElementById('welcome') .innerHTML = name; // DANGER
Vulnerable result appears here.
// SAFE CODE const name = location.hash.slice(6); document.getElementById('welcome') .textContent = name; // SAFE
Safe result appears here.
🔑 The fix is ONE WORD: change innerHTML to textContent. innerHTML tells the browser to parse and execute HTML. textContent tells it to display as plain text — always.
